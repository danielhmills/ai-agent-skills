The Semantic Medallion

The Semantic Medallion

Building a Knowledge Graph-Powered Data Catalog — augmenting traditional medallion architecture with RDF, DCAT, and SPARQL to create a self-describing catalog where metadata and data coexist in the same queryable graph.

By Veronika Heimsbakk · Knowledge Graph Specialist at Data Treehouse · Modern Data 101 · May 14, 2026

Overview

From Raw Data to Knowledge Graph

The Semantic Medallion redefines the traditional Bronze→Silver→Gold data pipeline by making the Gold layer an RDF knowledge graph. DCAT catalog metadata is woven into the same graph, creating a unified query surface where data catalog and customer facts coexist.

Architecture

The Three Medallion Layers

Bronze
Raw Harvest
Silver
IRI Minting
Gold
RDF Knowledge Graph
B

Bronze Layer

Raw data harvested from source systems — CRM, ERP, billing, support. No semantics applied. This is pure raw material ingestion. Data is stored as-is, preserving original formats and timestamps.

Semantic Enrichment: None · Order: 1

S

Silver Layer

Every entity receives a stable globally unique IRI. Raw records become identifiable, linkable resources with local semantics. IRI minting is the critical step — without stable identifiers, cross-system linking breaks.

Semantic Enrichment: Local · IRI Minting: Yes · Order: 2

G

Gold Layer

DataFrames are mapped to RDF via a shared ontology using OTTR templates and maplib. Global semantics enable cross-system entity resolution, semantic search, and impact analysis through SPARQL — the traditional Gold layer reimagined as linked data.

Semantic Enrichment: Global · Order: 3

Technology Stack

Tools That Power the Semantic Medallion

maplib

Python library for DataFrame-to-RDF transformation. A single maplib.map(dataframe, template, result_container) call converts Silver-layer DataFrames into RDF triples using OTTR templates — reducing hundreds of lines of triple-construction code to four lines of Python.

OTTR (Reasonable Ontology Templates)

Template-based ontology mapping framework. stOTTR (string syntax) defines reusable templates that declaratively map DataFrame columns to RDF ontology classes and properties, enabling consistent repeatable transformations.

DCAT — Data Catalog Vocabulary

W3C standard for describing data catalogs, datasets, distributions, and data services in RDF. Applied at each medallion layer to weave catalog metadata into the knowledge graph — the catalog describes itself.

PROV-O

W3C Provenance Ontology for tracking data lineage. Answers "where did this record come from?" by linking Gold-layer entities back to their source systems — CRM, ERP, billing — via prov:wasDerivedFrom.

SHACL

Shapes Constraint Language — W3C standard for validating RDF graphs against shape conditions. Enforces data quality rules: every Customer must have required properties, contracts must have valid statuses, data sources must be properly typed.

IRI Minting

The process of assigning stable globally unique identifiers to every entity at the Silver layer. IRIs are the foundation of linked data — they make entities referenceable across systems without fragile key matching.

The Catalog Experience

SPARQL Query Examples

Three queries demonstrating how a knowledge graph-powered catalog replaces traditional catalog browsing. Each query targets the URIBurner SPARQL endpoint and is backed by sample instance data making them fully executable.

Query 1: All Facts About Customer C-001

Retrieve every property-value pair for a single customer. The simplest entry point into the catalog — equivalent to "show me everything we know about Acme Corporation."

PREFIX : <https://moderndata101.substack.com/p/the-semantic-medallion#>
PREFIX rdfs: <http://www.w3.org/2000/01/rdf-schema#>
PREFIX prov: <http://www.w3.org/ns/prov#>

SELECT ?property ?value WHERE {
  :C-001 ?property ?value .
  OPTIONAL { ?value rdfs:label ?label }
}
ORDER BY ?property

Query 2: Contributing Data Sources for Customer C-001

Trace the provenance chain — list all source systems (CRM, ERP, billing) that contributed records for a customer. Answers "where did this customer view come from?" using PROV-O lineage.

PREFIX : <https://moderndata101.substack.com/p/the-semantic-medallion#>
PREFIX prov: <http://www.w3.org/ns/prov#>

SELECT ?source ?sourceName ?sourceType WHERE {
  :C-001 prov:wasDerivedFrom ?source .
  ?source a :DataSource ;
    :sourceName ?sourceName ;
    :sourceType ?sourceType .
}
ORDER BY ?sourceName
▶ Run Live Query
Query 3: Customers with Billing Issues AND Active Contracts

Cross-domain pattern detection — find all customers who have both an open billing issue and an active contract. This query spans the billing, contract, and customer domains, a pattern that would require complex joins across siloed SQL databases but is natural in a knowledge graph.

PREFIX : <https://moderndata101.substack.com/p/the-semantic-medallion#>
PREFIX rdfs: <http://www.w3.org/2000/01/rdf-schema#>

SELECT ?customer ?name ?issueDescription ?contractStatus WHERE {
  ?customer a :Customer ;
    :hasName ?name ;
    :hasBillingIssue ?issue ;
    :hasContract ?contract .
  ?issue a :BillingIssue ;
    :issueDescription ?issueDescription .
  ?contract a :Contract ;
    :contractStatus "Active" .
}
ORDER BY ?name
▶ Run Live Query

Frequently Asked Questions

FAQ

What is the Semantic Medallion?

The Semantic Medallion is an augmentation of the traditional medallion data architecture (Bronze → Silver → Gold) where the Gold layer becomes an RDF knowledge graph instead of clean tabular data. DCAT catalog metadata is woven into the same graph, creating a self-describing knowledge graph-powered data catalog.

How does the Bronze layer differ from traditional medallion Bronze?

In the Semantic Medallion, the Bronze layer functions identically to traditional medallion — it harvests raw data from source systems without transformation. The difference emerges at Silver, where IRIs are minted instead of just cleaning data.

What is IRI minting and why is it important?

IRI minting is the process of assigning a stable globally unique Internationalized Resource Identifier to every entity at the Silver layer. It is critical because it makes entities referenceable across systems — without stable IRIs, linking customer records from CRM, ERP, and billing systems would require fragile key matching. IRIs are the foundation of linked data.

What role do OTTR templates play?

OTTR (Reasonable Ontology Templates) provides a declarative template language for mapping structured data to RDF. Using stOTTR syntax, templates define how DataFrame columns map to ontology classes and properties. A single template can transform millions of Silver-layer rows into RDF triples in a consistent repeatable way.

How does maplib simplify the DataFrame-to-RDF transformation?

maplib is a Python library that accepts a pandas DataFrame, an OTTR template, and a result RDF container, then executes the mapping in a single call. This reduces what would otherwise be hundreds of lines of triple-construction code to four lines of Python.

What is DCAT and why is it used in the Semantic Medallion?

DCAT (Data Catalog Vocabulary) is a W3C standard for describing data catalogs, datasets, distributions, and data services in RDF. In the Semantic Medallion, each medallion layer receives DCAT descriptions — the catalog metadata and the actual data coexist in the same graph, making the catalog self-describing rather than a separate metadata store.

How does PROV-O enhance the catalog?

PROV-O (Provenance Ontology) tracks data lineage — where each record came from, which system generated it, and what transformations were applied. In the catalog, provenance links connect customer facts back to their source systems, answering questions like "Which CRM and billing records contributed to this customer view?"

What kinds of queries does the knowledge graph catalog enable?

The knowledge graph catalog enables three categories of queries: entity lookup (retrieve all facts about a single entity), provenance tracing (list all source systems contributing to a record), and cross-domain pattern detection (find entities matching conditions spanning multiple domains, such as customers with both active contracts and open billing issues).

How does the Semantic Medallion handle entity resolution?

Entity resolution happens naturally through IRI alignment. Because every entity receives a globally unique IRI at the Silver layer, records from different source systems that refer to the same real-world entity are linked to the same IRI. The Gold layer's ontology then connects these IRIs across domains.

What is SHACL and how does it fit?

SHACL (Shapes Constraint Language) is a W3C standard for validating RDF data against shape conditions. In the Semantic Medallion architecture, SHACL shapes can be applied to the Gold layer to enforce data quality rules — ensuring every Customer has required properties, contracts have valid statuses, and data sources are properly typed.

What are the key lessons from implementing the Semantic Medallion?

Three key lessons: (1) Start with IRIs early — minting identifiers at the Silver layer pays compounding dividends as more systems connect. (2) Design ontologies iteratively — begin with a minimal shared ontology and expand as new use cases emerge. (3) The medallion is about semantic enrichment layers — Bronze has no semantics, Silver has local semantics, Gold has global semantics.

How does the Semantic Medallion compare to a traditional data catalog?

Traditional data catalogs describe where data lives (table names, column schemas, storage locations) but do not describe how data connects across systems. A knowledge graph-powered catalog captures both — DCAT metadata describes the datasets while RDF relationships connect customer entities to their source systems, contracts, and issues.

Glossary

Key Terminology

Medallion Architecture
A data architecture pattern organizing data into Bronze (raw), Silver (cleaned/enriched), and Gold (analytics-ready) layers. The Semantic Medallion redefines Gold as an RDF knowledge graph.
IRI Minting
The process of assigning stable globally unique Internationalized Resource Identifiers to entities, enabling cross-system linking and resolution in linked data architectures.
OTTR
A framework for template-based ontology mapping. stOTTR (string syntax) defines reusable templates that declaratively map structured data columns to RDF ontology classes and properties.
maplib
A Python library for DataFrame-to-RDF transformation using OTTR templates. Enables a single function call to convert pandas DataFrames into RDF knowledge graphs.
DCAT
A W3C standard RDF vocabulary for describing data catalogs, datasets, distributions, and data services. Used in the Semantic Medallion to weave catalog metadata into the knowledge graph.
PROV-O
A W3C standard ontology for representing provenance information — tracking who generated data, what sources it came from, and what transformations were applied.
SHACL
A W3C standard for validating RDF graphs against shape conditions, enabling data quality enforcement in knowledge graph systems.
Knowledge Graph
A graph-structured representation of data where entities are nodes connected by typed relationships, enabling semantic querying and inference beyond what relational databases support.
SPARQL
SPARQL Protocol and RDF Query Language — the W3C standard query language for RDF knowledge graphs, analogous to SQL for relational databases but designed for graph pattern matching.
Semantic Enrichment
The progressive addition of meaning to data as it moves through the medallion layers: none at Bronze, local at Silver (IRIs within a domain), global at Gold (cross-domain ontology linking).
Implementation Guide

How to Build a Semantic Medallion Pipeline

  1. Harvest Raw Data into the Bronze Layer

    Ingest raw data from all source systems — CRM, ERP, billing, support — into the Bronze layer. Apply no transformations. Preserve original formats and timestamps.

  2. Mint Stable IRIs at the Silver Layer

    For every entity in the Bronze data, create a stable globally unique IRI. Choose a consistent IRI pattern. This is the most critical step — IRIs are the foundation for all downstream linking.

  3. Design a Shared Ontology

    Define the core classes (Customer, Contract, DataSource) and properties (hasContract, wasDerivedFrom) that span your business domains. Start minimal and expand iteratively.

  4. Author OTTR Templates

    Write stOTTR templates that map Silver-layer DataFrame columns to your shared ontology. Each template defines how a row becomes RDF — which column maps to which class, which columns become properties.

  5. Transform to RDF with maplib

    Load Silver-layer DataFrames into pandas. Call maplib.map(dataframe, template, result_container) for each template. Validate the output RDF syntax before proceeding.

  6. Weave in DCAT and PROV-O Metadata

    Describe each medallion layer as a DCAT Dataset with distributions and data services. Add PROV-O provenance links connecting Gold-layer entities back to their source systems.

  7. Query and Validate

    Load the RDF into a SPARQL-capable triplestore. Run validation queries. Apply SHACL shapes for data quality enforcement. Then query your knowledge graph catalog — start with entity lookup, progress to provenance tracing, and finally cross-domain pattern detection.
