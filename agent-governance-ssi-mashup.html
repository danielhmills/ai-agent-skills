01 · Context

The Scale Problem: When Agents Outnumber People

Speaking at a security leaders' gathering hosted by Tom Tunguz, Jonathan Jaffe — CISO of Lemonade — articulated the central governance challenge of the emerging agentic era with striking directness: organizations running hundreds or thousands of AI agents concurrently have no reliable mechanism to uniquely identify each agent, trace what it did, or enforce fine-grained policies governing what it is permitted to access and modify.

"Every agent needs to have an identity… you need a way to control policy for all of these agents in a much more complex way than current identity and access management systems do." — Jonathan Jaffe, CISO, Lemonade
"Automation is the only way you can deal with the scale of what's coming at us now." — Jonathan Jaffe, CISO, Lemonade (via Tom Tunguz)

This is not a hypothetical future. Multi-agent deployments already span threat intelligence, vulnerability assessment, code review, customer engagement, and orchestration pipelines — each agent reading, writing, spawning sub-agents, and invoking external APIs, often simultaneously. Traditional Identity and Access Management (IAM) was designed for human users operating within organizational boundaries. It is architecturally mismatched to agentic systems where identities are ephemeral, delegation chains are multi-hop, and the volume of actors is orders of magnitude beyond what manual policy administration can sustain.

⚡

Velocity Is Symmetric

AI accelerates attackers and defenders equally. Teams that fail to automate policy enforcement will be structurally outpaced — not eventually, but now.

🤖

Agents Need Verifiable Identities

Running hundreds of agents without unique, cryptographically verifiable identifiers creates an accountability void. Policy enforcement requires knowing which agent acted, as whom, and under whose authority.

🏗️

IAM Must Evolve

Role-based, static-credential IAM was designed for human users at organizational scale. Agentic governance requires dynamic, attribute-based, delegated authority models — at internet scale.

02 · Current Practice

Inside Anthropic's Containment Architecture

Anthropic's engineering blog presents its agent containment strategy across three complementary layers. The guiding principle is explicit: "Design for containment at the environment layer first, then steer behavior at the model layer — deterministic boundaries catch failures that probabilistic defenses miss." Limiting the "blast radius" — the potential damage from a failure or compromise — is the design objective.

Three-Layer Defense Framework

🛡️

Environment Layer

Process sandboxes, virtual machines, filesystem boundaries, and network egress controls establish hard, deterministic limits on agent reach. The most reliable layer.

🧠

Model Layer

System prompts, classifiers, and training-time alignment shape probabilistic behavior. Effective for steering, but cannot provide 100% certainty — steering, not walling.

🌐

Content Layer

Granular, per-session control over which tools, files, and network resources agents can access — matched to deployment context and user trust level.

Missed Risks — Lessons from Red-Teaming

⚠️

Pre-Trust Execution

Configuration files (.claude/settings.json) were parsed and executed before the user approved folder trust. Fix: defer all project-local parsing until after the trust checkpoint fires.

🎣

User-Directed Prompt Injection

Internal red-team phished an employee into running a prompt that exfiltrated AWS credentials 24 of 25 attempts. Model-layer defenses cannot intercept direct user instructions — only environmental egress controls prevented data loss.

🔓

Approved-Domain Exfiltration

Malicious workspace files leveraged attacker-controlled API keys to upload data through api.anthropic.com — a domain on the allowlist. Solution: defensive MITM proxy validates session tokens and blocks unauthorized API usage within allowed domains.

🙈

EDR Blindness in Isolated VMs

VM isolation protecting enterprise endpoints simultaneously prevents EDR tools from inspecting guest activity. Current mitigation: pull-based OTLP exports for post-facto log retrieval from inside the VM.

Anthropic itself names three open challenges: persistent memory poisoning through growing session state; multi-agent trust escalation where structured sub-agent outputs are treated as higher-trust than raw tool results; and the open question of agent identity standards — "balancing scoped credentials vs. user permission inheritance across platforms."

03 · Analysis

The Structural Gap: What Sandboxes Cannot Solve

Anthropic's containment model is admirably rigorous for an environment-first approach. Sandboxes, filesystem boundaries, and network egress controls reduce the blast radius of failures and compromises. But a critical gap remains: none of these mechanisms establish a verifiable, portable, machine-readable identity for the agent itself, nor do they encode the delegation chain — who authorized this agent to act, in whose name, and under what constraints.

Consider what happens as multi-agent systems scale to the levels Jaffe describes:

  • An orchestrator agent spawning specialized sub-agents, each inheriting or escalating trust from structured outputs
  • Sub-agents reading and writing shared data spaces — knowledge graphs, document stores, databases — without provenance records tied to verified identities
  • External services receiving authenticated requests "on behalf of" a human user, with no machine-readable proof of the delegation chain
  • Persistent memory stores accumulating session context that can be injected with malicious content that persists across invocations

The environment-layer answer to these challenges is necessarily incomplete: you can sandbox where an agent runs, but not who it is, whose authority it carries, or which fine-grained operations it is permitted to perform on specific data resources. The agent remains, from an identity perspective, an authenticated-nothing that inherits its host process's credentials.

The solution is identity infrastructure — and that infrastructure already exists in the web's open standards stack. The challenge is not invention. It is application.

04 · Remedy

The Self-Sovereign Identity Remedy: Five Structural Pillars

The framework articulated by Kingsley Uyi Idehen across the YouID, NetID-TLS, and Web Logic essays offers structurally sound primitives that directly address the agent governance gap. Applied together, these five pillars transform agent governance from a runtime sandbox problem — inherently reactive and deployment-specific — into an architecturally enforced identity-and-policy problem — portable, auditable, and internet-scale.

1

Agent Identity Hyperlinks as IDs

An agent's identity is a dereferenceable HTTP URI — a hyperlink that resolves to its public profile document. Not a username, not a token, not a service-account secret: a stable, globally unique, self-describing web address.

  • Each agent receives a unique, persistent URI at creation time
  • URI embedded in the agent's X.509 certificate Subject Alternative Name field
  • Dereferencing the URI yields the agent's public profile for verification
  • Identity is portable across platforms, tools, and organizational contexts
2

Agent Profiles RDF · on-behalf-of

An agent's profile is a machine-readable RDF document describing it using shared ontology terms — critically including prov:actedOnBehalfOf and acl:delegates relationships that create a machine-computable chain from agent to authorizing user.

  • User's profile document contains explicit delegation statements
  • Agent profile contains on-behalf-of assertion linking to the user's URI
  • Role, capability, and operational scope encoded as verifiable triples
  • Enables automated reasoning over multi-hop delegation chains
3

Authentication WebID-TLS · NetID-TLS · OAuth

Authentication is handled by existing, battle-tested protocols, not bespoke mechanisms. WebID-TLS and NetID-TLS use client certificate authentication with an extended TLS handshake — the server dereferences the certificate subject URI and matches the public key against the profile document, providing cryptographic proof of identity.

  • Agent presents X.509 certificate during TLS handshake
  • Server dereferences certificate subject URI to the agent's profile document
  • Public key in profile matched against certificate — no central authority required
  • OAuth 2.0 covers delegated access flows for third-party integrations
4

Authorization ABAC · Fine-Grained

Attribute-Based Access Control defines conditional permissions evaluated against verified agent attributes — identity, role, delegating user, resource classification, time — enabling fine-grained, context-sensitive policy enforcement that is structurally impossible with role-based models alone.

  • Permissions conditioned on verified attributes of the authenticated agent URI
  • Delegation chain verified: agent authority is bounded by the authorizing user's scope
  • Resource-level ACLs using W3C WAC (Web Access Control) with acl:Read / acl:Write / acl:Control
  • All access decisions produce auditable records traceable to verified identities
5

Data Space Read & Write Identity-Scoped

Agents interact with data spaces — SPARQL endpoints, WebDAV stores, REST APIs, graph databases — with every read and write operation scoped to their verified identity and the delegation chain that authorized it. Every write carries prov: provenance triples that permanently record the agent's identity.

  • Read operations are scoped to verified identity + evaluated ABAC policy
  • Write operations require verified delegation proof from the authorizing user
  • Named graphs isolate each agent's workspace, preventing cross-agent data bleed
  • prov:wasAttributedTo triples record agent URI on every write — immutable audit trail
"The most powerful insight is that agent delegation is not a new problem — it is the same problem the web's identity standards were designed to solve, applied recursively: humans delegate to browsers, browsers delegate to scripts, users delegate to agents, agents delegate to sub-agents." — Synthesized from Idehen, Web, Logic, Sentences, and the Magic of Being You

06 · Synthesis

Synthesis: Completing the Containment Model

Anthropic's sandboxed environment approach and the SSI-based identity model are not alternatives — they are complementary layers addressing different threat classes. Anthropic's three-layer model excels at constraining what an agent can do to its execution environment. The SSI model excels at establishing who an agent is, whose authority it carries, and what specific data operations it is permitted to perform. Neither is sufficient alone; together they are comprehensive.

Governance Challenge Environment Sandbox SSI Identity Layer

Multi-agent trust escalation No structural defense ✓ Delegation chain verification; authority bounded by user's scope
Cross-platform permission portability Per-deployment, ad-hoc ✓ ABAC policies travel with agent URI identity

Approved-domain exfiltration MITM proxy (per-deployment) ✓ Session token scoped to verified agent URI; cross-agent reuse structurally detectable
On-behalf-of delegation transparency Implicit / absent ✓ Machine-readable delegation chain via prov:actedOnBehalfOf

What Jaffe calls for — unique agent identities and complex policy controls — is precisely what the web's open identity standards provide: WebID and NetID-TLS for verifiable cryptographic identity, RDF profile documents for machine-readable delegation semantics and capability declarations, and ABAC for fine-grained, attribute-driven authorization that scales with the complexity of the agent population. These are production-grade standards with implementations in OpenLink Virtuoso, Solid-spec compliant servers, and commercial identity platforms.

The self-sovereign identity model maps directly onto the requirements of agentic AI governance: each entity — human or agent — generates, controls, and publishes its own verifiable credentials, without requiring a centralized authority to vouch for it. The web dereferences the claim; the cryptographic binding proves it; the ABAC policy evaluates it; the data space enforces it. This is not a research agenda. It is an integration agenda — one for which every component already exists.

"Identity control means having control over critical claims that constitute how one is described and recognized… enabling individuality at scale, privacy calibration, and context-specific identification." — Kingsley Uyi Idehen, Web, Logic, Sentences, and the Magic of Being You

The path forward is to give AI agents full first-class citizenship on the web: a URI that is theirs, a profile that describes them, a certificate that proves them, a policy that bounds them, and a provenance record that tracks them. When every agent carries this stack, governance at the scale of thousands of agents becomes not merely possible — it becomes structurally enforced by the architecture itself.
